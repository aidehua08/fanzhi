# 泛知科技高质量首页重做规格：AI 落地指挥舱

## 背景判断

当前首页已经具备基本结构和 3D 氛围，但首屏仍偏“数据背景”，缺少让客户第一眼相信“泛知科技真的懂 AI 落地”的强演示感。用户明确反馈：首页是转化主战场，不能先将就，应该达到 Remix 和 Sazabi 参考站那种更高技术含量、更真实、更震撼的第一印象。

新版首页不再把 3D 视为装饰背景，而是把首屏做成一个“企业 AI 落地指挥舱”的实时演示：客户看到的不是抽象球体，而是一个 AI 系统正在诊断企业、生成培训路径、拆解 Agent 落地任务。

## 目标

首屏的单一目标：让中小企业老板在 5 秒内形成三个感受：

- 泛知科技不是卖课，而是做企业 AI 落地系统。
- 这家公司技术感强，懂咨询、培训、Agent、知识库和陪跑的完整链路。
- 点击“预约 AI 诊断”是自然下一步。

首页后续内容可以继续完善，但首屏必须先达到“可信、专业、有冲击力”的门面效果。

## 参考站拆解

### Remix 参考点

Remix 的高级感来自发布会式首屏：大标题、强空间感、精确的动效节奏、清晰的品牌姿态。它适合借鉴：

- 巨型标题带来的压迫感。
- 背景动效和标题节奏统一。
- 首屏像一个完整品牌事件，而不是普通网页区块。

不直接复制 Remix 的原因：Remix 叙事偏开发框架，泛知科技面向中小企业老板，不能只做抽象技术美感。

### Sazabi 参考点

Sazabi 的高级感来自“真实系统正在运行”：页面不是展示一个漂亮背景，而是展示 AI-native observability 的产品能力。它适合借鉴：

- 首屏即产品演示。
- 系统 UI、告警、聊天、Agent 状态共同形成真实感。
- 未来感来自“业务流动起来”，不是来自单纯发光粒子。

不直接复制 Sazabi 的原因：Sazabi 面向工程 observability，泛知科技要表达企业经营、咨询、培训和落地陪跑。

## 推荐方向

采用融合方向：**真实 AI 经营舱 + 发布会级 3D 氛围**。

首屏画面不是一个普通 3D 球，而是一个正在运行的企业 AI 系统：

- 左侧或中部有强标题，保持 Remix 式压迫感。
- 右侧或中心是一个可视化 AI 经营舱，保持 Sazabi 式真实系统感。
- 背景是高质量 WebGL 空间，提供未来感，但不抢业务表达。
- 动效围绕“企业痛点被 AI 识别并转化为落地任务”展开。

## 首屏叙事

首屏需要演示一个完整业务闭环：

1. 企业痛点进入系统  
   示例输入：`客服响应慢 / 销售跟进断层 / SOP 难执行 / 内部知识散落`

2. AI 诊断生成痛点地图  
   系统自动把痛点归类到咨询、培训、知识库、Agent、自动化。

3. 生成三阶段落地路径  
   阶段一：AI 咨询诊断  
   阶段二：企业培训共识  
   阶段三：Agent / 知识库 / 自动化陪跑

4. 预约诊断 CTA 出现  
   用户理解泛知科技的交付方式后，主按钮成为自然动作。

- 动效保留扫描、打字、状态流转，不做大量 3D 视角旋转。

## 关键动效

### 页面进入

1. 黑场或深色空间加载。
2. 3D 背景出现轻微景深和扫描光。
3. 指挥舱面板从模糊到清晰。
4. 痛点输入逐字出现。
5. AI 诊断节点依次点亮。
6. 主标题和 CTA 稳定落位。

### 鼠标互动

- 鼠标移动时，背景空间有低幅视差。
- 指挥舱面板按层级轻微偏移，形成真实深度。
- 鼠标靠近指挥舱时，模块边缘发光，像系统可交互。
- 点击页面触发一次局部扫描波，不做廉价大水波。

### 指挥舱自动循环

自动循环一段 12-16 秒的业务演示：

1. 痛点输入：`销售线索跟进断层`
2. AI 归因：`流程缺口 / 话术不统一 / 数据未沉淀`
3. 方案生成：`销售跟进 Agent + 话术知识库 + 培训工作坊`
4. 状态推进：`诊断完成 -> 培训排期 -> Agent 原型 -> 陪跑上线`

循环要像真实产品状态变化，不像普通轮播。

## 技术路线

### 必选技术

- React + TypeScript 延续现有项目。
- Three.js 保留做 WebGL 背景空间。
- Framer Motion 保留做 DOM/HUD 动效。
- 新增 GSAP，用于首屏时间线编排。
- 可新增 `postprocessing` 或 Three.js 内置 UnrealBloomPass，实现更像高端官网的 bloom、辉光和景深。

### 可选技术

- React Three Fiber：如果要把 3D 场景组件化，可引入。但首版不强制，避免一次性改动过大。
- Lottie/Rive：如果后续需要更精细的图形状态机，可作为第二阶段。
- 开源模板参考：可以参考 Remix 网站源码结构，但不复制站点视觉和代码。

### 不继续沿用的实现

- 不再把 3D 数据球作为首屏主角。
- 不再依赖大量粒子、圆环、网格制造科技感。
- 不再把“不卡”作为牺牲质感的理由；性能通过分层和降级解决。

## 组件架构

保留现有 hero 目录，但重构职责：

- `HeroCockpit`：首屏容器、指针状态、滚动入口。
- `CockpitCanvas`：WebGL 背景空间、镜头视差、光场、低成本粒子。
- `HeroHud`：品牌、主标题、CTA、关键信任信息。
- `AIDemoConsole`：新增，承载真实系统演示。
- `DiagnosisStream`：新增，逐步显示企业痛点和 AI 诊断。
- `DeliveryTimeline`：新增，显示咨询、培训、Agent、陪跑的阶段推进。
- `PointerAura`：保留，但从大光晕改成更克制的扫描准星。

## 内容模块

指挥舱内展示四个核心模块：

1. 痛点识别  
   展示企业问题输入和 AI 自动分类。

2. 落地路径  
   展示咨询、培训、Agent、知识库、陪跑。

3. Agent 队列  
   展示销售助手、客服助手、SOP 助手、老板经营助手等场景。

4. 实施状态  
   展示 `诊断中`、`方案生成`、`原型开发`、`陪跑上线`。

## 视觉风格

颜色继续保持深色科技，但提升质感：

- 背景：`#030712`、`#07111F`
- 玻璃面板：`rgba(8, 16, 30, 0.72)`
- 主发光：`#5EEAD4`
- 系统蓝：`#38BDF8`
- 行动红：`#FF4D6D`
- 可信白：`#E5FAFF`
- 分割线：`rgba(148, 163, 184, 0.16)`

字体：

- 中文主标题使用系统黑体高字重，必须强而清晰。
- 英文系统标签使用等宽或窄体，保持 HUD 感。
- 不使用过度花哨字体，避免中小企业客户觉得像娱乐项目。

质感关键词：

- 企业级
- 真实系统
- 高端 WebGL
- 可执行
- 冷静但震撼

## 性能策略

高质量不等于无上限堆效果。性能策略如下：

- WebGL 背景只做空间、光场和少量粒子。
- 真实业务演示主要用 DOM/CSS/GSAP，保证清晰可读。
- 桌面端启用 bloom/视差/自动循环。
- 移动端关闭后处理和过多粒子，只保留核心演示。
- 尊重 `prefers-reduced-motion`。
- 首屏如果包体积过大，第二阶段做动态 import，把 3D 场景拆成懒加载 chunk。

## 验收标准

视觉验收：

- 第一眼不再像普通数据背景或低成本 3D 球。
- 首屏能看出“AI 正在帮企业解决问题”。
- 有发布会级压迫感，也有真实系统感。
- 不出现旧 logo、门头、门牌素材。
- 不出现元素重叠、按钮溢出、文字不可读。

业务验收：

- 用户能看懂泛知科技做三件事：咨询诊断、企业培训、Agent/知识库落地陪跑。
- CTA 出现时不突兀，用户知道为什么要预约诊断。
- 首页可以独立承担转化，不依赖用户继续看案例。

技术验收：

- `npm run build` 通过。
- `npm run lint` 通过。
- 本地 `http://127.0.0.1:5173/` 可访问。
- Playwright 截图验证桌面和移动端。
- 浏览器控制台没有运行错误。

## 实施边界

第一轮只重做首页首屏，不动后续业务区块结构。后续服务、案例、联系区可在首屏定稿后再优化。

第一轮可以保留当前品牌缩写 `FZ`，但页面显著品牌必须是 `泛知科技`。

CTA 暂时跳转到现有联系区或二维码占位；飞书表单和真实微信二维码在下一阶段接入。

## 推荐实施顺序

1. 重构首屏信息架构和中文主标题。
2. 新建 `AIDemoConsole`，完成真实 AI 落地系统演示。
3. 调整 `CockpitCanvas`，从主视觉球体改为高级背景空间。
4. 用 GSAP 编排进入动画和演示循环。
5. 做桌面端截图自检。
6. 做移动端降级和截图自检。
7. 构建、lint、提交、推送。

## 决策

采用融合方向：真实 AI 经营舱为主，发布会级 3D 氛围为辅。

不继续微调当前数据球首屏。当前版本可作为历史草稿保留在 git 中，但下一版实现应重构首屏，而不是在现有球体上继续加装饰。
